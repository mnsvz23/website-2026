Case Study — ExoWork, 2026

Taming the Vibe

While AI-generated code lets you ship faster and get a product in the hands of customers quickly, it also compounds UX debt in ways that only show up when real people try to use the thing.

Jason Taming the Bulls of Aeëtes — Jean-François de Troy, 1742
Context

The product and my role

ExoWork is a co-pilot for agentic workflows, built at Realfast. It's an Electron Mac app that gives Salesforce consultants and BAs a way to work with AI coding agents without ever touching a terminal. Think file explorer, conversation UI, permission management, onboarding wizards. The kind of product layer that makes the underlying AI actually usable for people who aren't developers.

We shipped the initial version fast using AI-generated code. It worked, it was functional, and it got into the hands of early users quickly. But as consultants started using it day to day, the UX gaps became hard to ignore. Navigation was confusing. Permissions deadlocked the screen. Error messages read like stack traces.

I came in as design lead and had to onboard myself on the same AI tech stack that the engineers were using. There was no Figma handoff here. I was working directly on the live codebase, using Claude Code as my implementation partner, shipping design improvements straight to production. Over three months I put out 90 PRs across four repos.

Problems

What users ran into

Vibe-coded software has a specific failure mode. The code works, the features are all there, but there's no design thinking underneath. No information architecture, no awareness of who the user is, no consistency. The same kinds of problems showed up everywhere in ExoWork once real consultants started using it.

"I don't know where I am"

The sidebar mixed navigation with view-switching. Breadcrumbs skipped hierarchy levels. The Help page took over the entire navigation state and left users trapped with no back button. Every screen felt like a dead end.

"I don't understand what it's asking me"

Permission prompts showed labels like "Bash requires approval." The dismiss button silently denied permissions, permanently, with no way to undo it. Error messages showed raw codes like SDK_STREAM_ERROR. Onboarding tours talked about "tool calls" and "the agent."

"This doesn't work on my computer"

Keyboard shortcuts broke on Windows because the AI had assumed macOS key values. Hotkey labels showed Mac symbols on PCs. Light mode had 2.5:1 contrast on action buttons, which is basically invisible.

Approach

How I worked

The first couple of weeks were reactive. Users hit a problem, I fixed it. But by week three I noticed I was fixing the same categories of issues over and over. Permission UX, navigation confusion, developer-facing copy. Individual fixes weren't going to cut it.

So I shifted to treating these as design surfaces, not bug queues. I'd pick an area like "all user-facing copy" or "the entire permission flow" and sweep through it end to end. I wrote jargon-guard tests, 139 of them, that scan every string in the app for banned developer terms. If anyone introduces a tooltip that says "CLI" or "tool call," the build breaks.

The interesting part is that I was using the same AI tools that created these problems to fix them. Claude Code was my implementation partner throughout. The difference wasn't the tool. It was showing up with a clear design intent before starting a conversation with the AI. When I knew exactly what I wanted, the AI was great at getting me there fast.

Solutions

What I designed

Navigation

This one took three tries. The first pass removed breadcrumbs and added a persistent sidebar, which helped but still mixed "go to a place" with "change how you see this place." The second approach got opened as a PR, debated, and scrapped. Wrong model. The third attempt finally clicked: a collapsible left nav with three clear tabs (Workplan, Chats, Files), Help moved to a modal overlay, and a consistent back button on every screen.

Before: mixed sidebar with no hierarchy
After: three-tab collapsible left nav

Permission system

This was the surface I touched the most. 13 iterations over the full three months. The original was a full-width banner with developer jargon that could deadlock the entire screen if the backing API call failed. I redesigned it as a floating pill with three risk tiers (low, medium, high), plain-language labels, a dismiss that's purely visual with a restore option, and smart defaults that remember your preferences per project. Every round was shaped by watching real consultants interact with the prompts.

Video: permission bar in action, risk tiers, dismiss and restore

Copy and language

The AI had piped raw error codes straight to the screen. I replaced that with a central error map that translates machine errors into plain recovery instructions. I also built the jargon-guard test suite that catches developer language anywhere in the UI. "Checklist" became "Workplan." "Open Terminal" became "New Terminal." Every empty state got an actual call to action instead of a blank screen.

Before: "SDK_STREAM_ERROR"
After: "Something went wrong. Try again."

Platform compatibility

It took three separate passes just on keyboard shortcuts. First terminal hotkeys, then bracket and backtick keycodes that Chrome on Windows reports differently, then Chrome-reserved shortcut conflicts. Fixing these one by one was slow. The real unlock was stepping back and treating platform compatibility as a design category to audit end to end, not a bug list to work through.

After: platform-aware labels and keybindings

Reflection

What I learned

Vibe-coded UX breaks in predictable ways. No information architecture, no sense of who the user is, platform assumptions baked in from training data. Once you see the pattern, you can audit for it systematically instead of waiting for bug reports.

Fixing things after the fact doesn't scale. I touched the permission bar 13 times. Each round taught me something, but it also made clear that design rules need to live where the AI works, not in a separate tool. That insight became the basis for ExoDesigner, the design system I built next.

AI is a great collaborator when you bring the intent. I used the same tools the engineers use. What made the difference was knowing what I wanted before I opened a conversation with the AI. Clear design intent in, good implementation out.
