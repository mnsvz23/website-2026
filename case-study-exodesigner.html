Case Study — ExoDesigner, 2026

ExoDesigner — A design reviewer on every branch

After three months of fixing vibe-coded UX by hand, I built a design system that lives inside the AI itself. It enforces standards at generation time, not review time.

Context

Why I built this

During my work on ExoWork at Realfast, I shipped 90 PRs fixing UX problems that AI-generated code had created. I touched the permission bar 13 times. Redesigned the navigation 3 times. Rewrote the same categories of copy over and over. The fixes worked, but the process didn't scale.

The core issue was that traditional design systems don't reach the AI. A Figma library or a style guide doc assumes a human reads it and applies the rules during implementation. But when Claude Code is generating the components, nobody reads the style guide. The AI just produces functional code with hardcoded hex values, inconsistent spacing, and developer-facing copy.

I needed a design system that lives where the AI already works: in the codebase, in the terminal, at the moment the code is being written.

Problem

Why design systems fail with AI

After spending three months on ExoWork, I had a pretty clear picture of how vibe-coded UX breaks. It's not random. The AI makes the same category of mistake repeatedly, and you end up fixing the same things over and over.

Review-time gates don't scale

When AI can produce a full feature in minutes, post-hoc design review becomes a bottleneck. Teams route around it. The system needs to constrain at generation time, not catch problems after the fact.

Figma doesn't reach the AI

A design system in a separate tool is invisible to the code generator. If it's not in the terminal, in the codebase, or in the prompt, it doesn't exist as far as the AI is concerned.

Violations are systematic

Hardcoded color values, missing design tokens, developer-facing copy, platform assumptions. The AI doesn't make random errors. It makes the same kinds of errors, which means you can write rules to catch them.

Every fix is a design rule waiting to be formalised

Each of the 90 PRs I shipped on ExoWork encoded some piece of design knowledge. The fix history is basically the specification for the design system, if you take the time to extract it.

Solution

A design system that runs as code

ExoDesigner is a set of Claude Code slash commands that scan, enforce, and generate design-compliant code. When you (or the AI) type /design-audit, the system scans the codebase across six categories: visual hierarchy, interaction patterns, component consistency, form design, polish, and responsive behavior. It returns a report with specific file paths and suggested fixes.

/design-fix auto-corrects what it can. It replaces hardcoded hex values with design tokens, fixes spacing inconsistencies, and adds missing accessibility attributes. /design-init detects the project framework (React, Salesforce LWC, or Rails) and generates the right theme config and CSS variables.

The key architectural decision was to have the design system write rules into CLAUDE.md, the file that Claude Code reads before every response. This means the AI follows design constraints not because someone told it to, but because the constraints are part of its context. The design system becomes the AI's design instinct.

Architecture

How it works

The system is TypeScript, modular, and test-covered. Analyzers scan the codebase and produce structured violation reports. Fixers auto-correct what's deterministic. Generators produce project-specific configuration: CSS variables, CLAUDE.md design rules, and compliance reports. Framework detection is automatic, so the same commands adapt their output for React with Tailwind versus Salesforce LWC with SLDS.

Toolkit

Eight commands

/design-init

Framework detection

Detects React, LWC, or Rails. Generates theme config, CSS variables, and CLAUDE.md rules.

/design-audit

Six-category audit

Visual hierarchy, interactions, components, forms, polish, responsive. Returns file paths and fixes.

/design-fix

Auto-correction

Replaces hardcoded colors with tokens, fixes spacing, adds accessibility attributes.

/design-check

Quick compliance

Fast pass/fail before commit. Catches regressions in spacing, color, and accessibility.

/design-theme

Theme generation

Brand palette with semantic roles. Primary, secondary, accent with automatic dark/light variants.

/design-report

Compliance report

Full audit output with severity levels, grouped by category. Exportable and trackable over time.

/design-generate

Component generation

New components that conform to the system from the start. No post-hoc cleanup needed.

/design-redesign

Guided redesign

Take an existing screen and redesign it within the system's constraints.

Impact

What changes

The shift is from catching design problems after the code is written to preventing them while the code is being generated. It's the difference between a review gate and a generation constraint.

Without ExoDesigner

  • AI generates components with hardcoded #362CFF
  • Spacing is a mix of 12px, 15px, 14px in one view
  • No ARIA labels on interactive elements
  • Light mode contrast at 2.5:1 on buttons
  • Error text: SDK_STREAM_ERROR
  • Design review catches issues after the PR
  • Same fix applied 13 times over 3 months

With ExoDesigner

  • AI uses var(--color-primary) from CLAUDE.md
  • 4px spacing grid enforced by design-check
  • Accessibility fixer adds ARIA attributes automatically
  • Contrast validated against WCAG AA at scan time
  • Copy rules in CLAUDE.md prevent jargon
  • Violations caught before the code exists
  • Fix encoded once, enforced forever

Reflection

What I learned

The design system has to live where the AI works. A Figma file is for the designer. CLAUDE.md is for the machine. Both are necessary, but only one of them scales when the AI is writing the code.

One person can't audit every AI-generated component forever. The 90 PRs I shipped on ExoWork were necessary, but that approach doesn't hold up over time. A system that runs every time, doesn't get tired, and doesn't skip the check because the deadline is tomorrow, that's what actually works.

Fix history is the spec. Every PR I shipped during ExoWork was a design rule waiting to be formalised. ExoDesigner didn't come from a design exercise. It came from three months of seeing the same problems and writing down how to prevent them.
